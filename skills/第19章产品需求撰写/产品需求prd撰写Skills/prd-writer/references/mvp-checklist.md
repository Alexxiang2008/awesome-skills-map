# MVP 功能筛选清单

## 核心原则

MVP (Minimum Viable Product) 是能够验证核心产品假设的最小功能集。

**关键问题**：没有这个功能，产品还能验证核心假设吗？

- 如果**能** → 放到 V2 及以后
- 如果**不能** → MVP 必须包含

---

## 筛选框架

### 1. 核心价值测试

| 问题 | 判断 |
|------|------|
| 这个功能直接解决用户的核心痛点吗？ | 是 → 可能是 MVP |
| 没有这个功能，用户还会使用产品吗？ | 否 → 必须是 MVP |
| 这个功能是"锦上添花"还是"雪中送炭"？ | 雪中送炭 → MVP |

### 2. 验证假设测试

| 问题 | 判断 |
|------|------|
| 这个功能能帮助验证产品的核心假设吗？ | 是 → MVP 候选 |
| 没有这个功能的数据，我们能判断产品方向吗？ | 否 → MVP 必须 |

### 3. 用户流程测试

| 问题 | 判断 |
|------|------|
| 用户能完成核心任务流程吗？ | 能 → 非 MVP |
| 这个功能是核心流程的必经环节吗？ | 是 → MVP 必须 |

---

## 常见陷阱

### ❌ 应该砍掉的功能

1. **用户设置页面**（MVP 可以硬编码默认值）
2. **多语言支持**（先做单一语言）
3. **复杂的权限系统**（MVP 可以只有一种角色）
4. **数据导出功能**（手动导出即可）
5. **通知系统**（用户主动查看即可）
6. **社交分享**（验证核心价值后再加）
7. **深度定制化**（先用固定配置）

### ✅ 通常必须保留的功能

1. **核心业务流程**（产品存在的意义）
2. **基本的用户认证**（如果需要识别用户）
3. **关键数据的增删改查**（核心数据操作）
4. **基本的错误处理**（不能让用户看到崩溃）

## 引导话术

当用户想把太多功能放进 MVP 时：

> "这个功能确实很棒！但我想问一下：如果我们先上线一个没有这个功能的版本，用户还能完成核心任务吗？如果可以，我建议我们把它放到 V2，这样可以更快地验证核心假设。"

> "我理解这个功能的价值，但从 MVP 的角度考虑，我们的目标是用最小成本验证产品方向。这个功能是验证方向必须的吗？还是说它是让产品更好用的锦上添花？"

> "让我们用一个简单的标准来判断：如果只有一周时间上线，这个功能还会保留吗？如果不会，它可能更适合放在 V2。"

## 功能优先级矩阵

```
高价值 ─────────────────────────────────────────
       │                    │                   │
       │   MVP 必须有       │   V2 优先         │
       │   (核心功能)       │   (高价值增强)     │
       │                    │                   │
       ├────────────────────┼───────────────────┤
       │                    │                   │
       │   谨慎考虑         │   V3+ 或放弃      │
       │   (可能是伪需求)    │   (nice to have)  │
       │                    │                   │
低价值 ─────────────────────────────────────────
       低复杂度                            高复杂度
```

## 检查清单模板

在确定 MVP 功能列表时，对每个功能进行以下检查：

```markdown
### 功能：[功能名称]

- [ ] 直接解决用户核心痛点？
- [ ] 是核心流程的必经环节？
- [ ] 没有它产品无法验证核心假设？
- [ ] 不是"锦上添花"类功能？
- [ ] 实现复杂度可接受？

**结论**：[ ] MVP / [ ] V2 / [ ] V3+
**理由**：[一句话说明]
```
